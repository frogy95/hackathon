# PRD: AI-Native 해커톤 평가 시스템

## 1. 개요

### 1.1 프로젝트 목적
해커톤 참가자가 직접 산출물을 제출하고, 관리자가 제출된 목록을 기반으로 AI 자동 평가를 실행하는 웹 애플리케이션. 제출부터 평가, 결과 확인까지의 전체 흐름을 하나의 시스템에서 처리한다.

### 1.2 배경
"Claude Code를 활용한 AI-Native 개발 입문" 해커톤에서 다수의 팀 산출물을 일관된 기준으로 평가해야 하는 니즈가 있다. 심사위원 간 편차를 줄이고, 평가 근거를 투명하게 제공하기 위해 AI 기반 자동 평가 시스템을 구축한다.

### 1.3 사용자 유형

| 사용자 | 역할 | 인증 |
|--------|------|------|
| 참가자 | 산출물 제출, 본인 결과 확인 | 이름 + 사번 기반 (별도 회원가입 없음) |
| 관리자 | 평가 세션 관리, 평가 실행, 전체 결과 조회 | 관리자 비밀번호 |

### 1.4 핵심 가치
- **자기 제출 방식**: 참가자가 직접 정보를 입력하여 관리자 부담 최소화
- **일관성**: 모든 프로젝트를 동일한 기준으로 평가
- **투명성**: 점수뿐 아니라 평가 근거(코드 인용, 파일 참조)를 함께 제공
- **효율성**: 수작업 대비 평가 소요 시간 대폭 단축

## 2. 기능 요구사항

### 2.1 평가 세션 관리 (관리자)

**FR-001. 평가 세션 생성**
- 관리자가 새로운 평가 세션을 생성할 수 있다.
- 세션 정보: 세션명(예: "2026 봄 해커톤"), 제출 마감일시, 안내 문구
- 세션 생성 시 고유한 제출 URL이 자동 발급된다. (예: `/submit/{session_id}`)

**FR-002. 제출 마감 관리**
- 마감일시 이전까지만 참가자 제출을 허용한다.
- 관리자가 마감일시를 수동으로 연장하거나 즉시 마감할 수 있다.

**FR-003. 평가 기준 설정**
- 기본 평가 기준(4개 대항목 + 배포 보너스)이 사전 정의되어 있다.
- 관리자가 평가 기준의 배점, 세부 항목, 루브릭을 수정할 수 있다.

### 2.2 참가자 제출 (참가자)

**FR-004. 산출물 제출 폼**
- 참가자가 제출 URL에 접속하면 다음 정보를 입력하는 웹 폼이 표시된다:

| 필드 | 타입 | 필수 | 검증 |
|------|------|------|------|
| 이름 | 텍스트 | O | 2자 이상 |
| 사번 | 텍스트 | O | 형식 검증 (설정 가능) |
| GitHub 저장소 URL | URL | O | 유효한 GitHub public repo URL |
| 배포 URL | URL | X | URL 형식 검증, 접근 가능 여부 사전 체크 |

- 제출 시 GitHub 저장소 URL의 유효성을 실시간 검증한다 (존재 여부, public 여부).
- 동일 사번으로 중복 제출 시 기존 제출을 덮어쓰기한다 (최종 제출 기준).
- 제출 완료 시 확인 화면을 표시한다 (제출 시각, 입력 내용 요약).

**FR-005. 제출 내역 확인**
- 참가자가 이름 + 사번으로 본인의 제출 내역을 조회할 수 있다.
- 마감 전이라면 수정·재제출이 가능하다.

### 2.3 제출 목록 관리 (관리자)

**FR-006. 제출 현황 대시보드**
- 관리자가 전체 제출 목록을 확인할 수 있다.
- 표시 정보: 이름, 사번, 저장소 URL, 배포 URL 유무, 제출 시각, 평가 상태
- 제출 상태별 필터링: 전체 / 제출완료 / 평가중 / 평가완료 / 오류

**FR-007. 제출 건 관리**
- 관리자가 특정 제출 건을 평가 대상에서 제외하거나 복원할 수 있다.
- 부적합한 제출(잘못된 URL 등)에 대해 메모를 남길 수 있다.

### 2.4 데이터 수집 및 AI 평가

**FR-008. 저장소 데이터 수집**
- 관리자가 "평가 시작" 버튼을 누르면, 제출된 저장소 목록에 대해 순차적으로 데이터를 수집한다.
- GitHub API를 통해 각 저장소의 다음 데이터를 수집한다:
  - 파일/폴더 구조 (tree)
  - README.md 및 주요 문서 파일 내용 (PRD, CLAUDE.md, .cursorrules 등)
  - 주요 소스 코드 파일 내용
  - package.json, requirements.txt 등 의존성 파일
  - 커밋 히스토리 요약 (커밋 수, 최근 커밋 메시지)
- Rate limit 초과 시 대기 후 재시도하며, 실패한 저장소는 별도 표시한다.

**FR-009. 배포 URL 스크린샷 수집**
- 배포 URL이 제공된 프로젝트는 Puppeteer(또는 Playwright)를 사용하여 자동 스크린샷을 캡처한다.
- 캡처 대상: 메인 페이지 (데스크톱 1280×800, 모바일 390×844 두 가지 뷰포트)
- 페이지 로드 후 3초 대기하여 동적 렌더링 완료를 보장한다.
- 배포 URL 접근 실패(404, 타임아웃 등) 시 배포 보너스 0점 처리하고 사유를 기록한다.
- 캡처된 스크린샷은 Claude API의 vision 입력으로 전달하여 시각적 평가에 활용한다.

**FR-010. AI 평가 실행**
- 수집된 저장소 데이터를 기반으로 Claude API를 호출하여 평가를 수행한다.
- 각 평가 항목별로 점수(정수)와 평가 근거(텍스트)를 생성한다.
- 평가 근거에는 구체적인 파일명, 코드 위치, 문서 내용에 대한 참조를 포함한다.
- 전체 진행률을 실시간으로 표시한다 (예: "12/30 평가 완료").

**FR-011. 평가 기준 (기본값)**

| 대항목 | 배점 | 세부 항목 |
|--------|------|-----------|
| AI-Native 문서화 체계 | 35점 | 프로젝트 정의 문서(15), AI 컨텍스트 파일(10), 진행 기록(10) |
| 기술 구현력 | 25점 | 아키텍처 설계(10), 코드 품질(8), 기술 스택 활용(7) |
| 완성도 및 UX | 25점 | 동작 완성도(12), 사용자 경험(8), 반응형/접근성(5) |
| 아이디어 및 활용 가치 | 15점 | 문제 정의(7), 차별성/실용성(8) |
| **배포 보너스** | **+10점** | 배포 가점(3), 시각적 완성도(7) |

> **총점 체계**: 기본 100점 + 배포 보너스 최대 10점 = **110점 만점**
> 배포 URL이 없는 프로젝트는 코드 기반 평가(100점 만점)로만 채점된다.

**FR-011-1. 배포 보너스 평가 기준 (최대 +10점)**

배포 URL이 제공되고 정상 접근 가능한 프로젝트에 한해 추가 평가한다.

| 세부 항목 | 배점 | 평가 기준 |
|-----------|------|-----------|
| 배포 가점 | 3점 | 배포 URL이 존재하고 정상 접근 가능하면 3점 만점 부여 (pass/fail) |
| 시각적 완성도 | 7점 | 스크린샷 기반 Claude Vision 평가: 레이아웃 정돈(2), 색상/타이포그래피 조화(2), 시각적 계층 구조(1), 모바일 대응(2) |

> 시각적 완성도 평가는 캡처된 데스크톱·모바일 스크린샷 이미지를 Claude API에 vision 입력으로 전달하여 수행한다.

### 2.5 결과 대시보드

**FR-012. 전체 순위 보기 (관리자)**
- 모든 평가 대상 프로젝트의 총점 기준 순위표를 표시한다.
- 항목별 점수를 함께 보여주며, 항목별 정렬이 가능하다.
- 배포 보너스 포함/미포함 순위를 토글로 전환할 수 있다.

**FR-013. 개별 프로젝트 상세 리포트 (관리자)**
- 프로젝트별로 항목별 점수, 세부 항목 점수, 평가 근거를 확인할 수 있다.
- 레이더 차트로 항목별 균형을 시각화한다.
- 배포 URL이 있는 프로젝트는 캡처된 스크린샷도 함께 표시한다.

**FR-014. 참가자 결과 확인**
- 참가자가 이름 + 사번으로 본인의 평가 결과를 조회할 수 있다.
- 관리자가 "결과 공개" 토글을 활성화한 이후에만 조회 가능하다.
- 참가자에게는 본인의 점수와 평가 근거만 표시된다 (타인 점수, 순위는 비공개).

**FR-015. 결과 내보내기 (관리자)**
- 전체 평가 결과를 CSV 또는 PDF로 내보낼 수 있다.

### 2.6 행운상 추첨 (관리자)

**FR-016. 행운상 랜덤 추첨**
- 관리자가 제출 완료된 참가자 중에서 행운상 당첨자를 랜덤 추첨할 수 있다.
- 추첨 설정: 당첨 인원 수, 추첨 대상 범위(전체 제출자 / 평가 대상자만 / 점수 하위 N% 등)
- 이미 실력 기반 수상자로 선정된 참가자를 추첨 대상에서 제외하는 옵션을 제공한다.
- 추첨 과정을 시각적 애니메이션(슬롯머신 또는 룰렛 방식)으로 연출하여 발표 현장에서 활용할 수 있다.
- 추첨 결과(당첨자 이름, 사번)를 기록하고, 필요 시 재추첨이 가능하다.

## 3. 비기능 요구사항

### 3.1 성능
- 단일 저장소 평가 완료: 180초 이내 (API 응답 시간 포함)
- 최대 100개 저장소 일괄 평가 지원
- 병렬 3개의 노드로 저장소 평가
- 일괄 평가 시 진행률 표시

### 3.2 보안
- 관리자 페이지는 비밀번호로 보호한다.
- 참가자 본인 확인은 이름 + 사번 조합으로 한다 (간이 인증).
- GitHub API 토큰은 서버 사이드에서만 사용하며 클라이언트에 노출하지 않는다.
- Claude API 키는 환경 변수로 관리한다.
- 평가 대상은 public repository로 한정한다.

### 3.3 에러 처리
- 저장소 접근 실패, API rate limit, Claude API 오류 시 사용자에게 명확한 피드백을 제공한다.
- 부분 실패 시 성공한 저장소의 결과는 보존한다.
- 개별 프로젝트 단위로 재평가가 가능하다.

## 4. 기술 스택 (제안)

| 구분 | 기술 | 선정 이유 |
|------|------|-----------|
| 프론트엔드 | Next.js (App Router) + TypeScript | 풀스택 단일 프레임워크, SSR 지원 |
| UI | Tailwind CSS + shadcn/ui | 빠른 UI 구축, 일관된 디자인 시스템 |
| 차트 | Recharts | 레이더 차트, 바 차트 등 시각화 |
| AI | Claude API (claude-sonnet-4-5-20250929) | 코드 분석 능력, 비용 효율 |
| GitHub 연동 | Octokit (GitHub REST API) | 공식 SDK, 안정성 |
| 스크린샷 캡처 | Puppeteer 또는 Playwright | 배포 URL 자동 스크린샷, 헤드리스 브라우저 |
| 데이터 저장 | SQLite (Drizzle ORM) | 별도 DB 서버 불필요, 로컬 실행 용이 |
| 배포 | Vercel 또는 로컬 실행 | 간편 배포 |

## 5. 핵심 데이터 모델

```
EvaluationSession (평가 세션)
├── id
├── name (예: "2026 봄 해커톤")
├── created_at
├── submission_deadline
├── description (안내 문구)
├── criteria_config (JSON - 평가 기준 설정)
├── results_published (boolean - 참가자 결과 공개 여부)
├── admin_password_hash
│
├── Submission (참가자 제출) [1:N]
│   ├── id
│   ├── session_id (FK)
│   ├── name (참가자 이름)
│   ├── employee_id (사번)
│   ├── repo_url
│   ├── deploy_url (nullable)
│   ├── submitted_at
│   ├── updated_at
│   ├── excluded (boolean - 관리자가 평가 대상에서 제외)
│   ├── admin_note (관리자 메모)
│   ├── status (submitted | collecting | evaluating | done | error)
│   ├── collected_data (JSON - 수집된 저장소 데이터)
│   ├── screenshots (JSON - 캡처된 스크린샷 경로 {desktop, mobile})
│   ├── total_score
│   ├── base_score
│   ├── bonus_score
│   │
│   └── Score (항목별 점수) [1:N]
│       ├── criteria_key
│       ├── score (정수)
│       └── reasoning (평가 근거 텍스트)
```

## 6. 사용자 흐름

### 6.1 관리자 흐름
```
[관리자 로그인]
    │
    ▼
[평가 세션 생성] → 제출 URL 발급
    │
    ▼
[제출 URL을 참가자에게 공유] (이메일, 슬랙 등)
    │
    ▼
[제출 현황 모니터링] ← 실시간 제출 목록 확인
    │
    ▼
[제출 마감]
    │
    ▼
[제출 목록 검토] → 부적합 건 제외 처리
    │
    ▼
[평가 실행] → GitHub 데이터 수집 → 스크린샷 캡처 → Claude API 평가
    │                                    (진행률 표시)
    ▼
[결과 대시보드] → 검토 후 "결과 공개" 활성화
    ├── 전체 순위표
    ├── 개별 상세 리포트
    └── CSV/PDF 내보내기
```

### 6.2 참가자 흐름
```
[제출 URL 접속]
    │
    ▼
[이름, 사번, GitHub URL, 배포 URL 입력]
    │
    ▼
[제출 완료 확인 화면]
    │
    ▼
(마감 전) [이름 + 사번으로 제출 내역 조회 → 수정/재제출]
    │
    ▼
(결과 공개 후) [이름 + 사번으로 본인 평가 결과 확인]
```

## 7. 주요 화면 구성

### 7.1 참가자 화면

**제출 폼 (`/submit/{session_id}`)**
- 세션명, 안내 문구 표시
- 입력 폼: 이름, 사번, GitHub URL, 배포 URL(선택)
- GitHub URL 입력 시 실시간 유효성 검증 (존재 여부, public 여부)
- 마감 시간 카운트다운 표시
- 제출 완료 후 확인 화면

**제출 확인 / 결과 조회 (`/check/{session_id}`)**
- 이름 + 사번 입력으로 본인 제출 내역 조회
- 마감 전: 수정 버튼 활성화
- 결과 공개 후: 항목별 점수 및 평가 근거 표시

### 7.2 관리자 화면

**대시보드 (`/admin`)**
- 평가 세션 목록, 세션 생성 버튼
- 세션별 제출 현황 요약 (제출 수, 평가 상태)

**세션 상세 (`/admin/session/{session_id}`)**
- 제출 목록 테이블 (정렬, 필터링, 검색)
- 평가 실행 버튼 + 진행률 표시
- 결과 공개 토글
- 순위표 + 개별 상세 리포트 링크
- 행운상 추첨 버튼

**행운상 추첨 (`/admin/session/{session_id}/lucky-draw`)**
- 추첨 설정: 당첨 인원 수, 제외 대상 선택
- 추첨 애니메이션 (프레젠테이션 모드 지원, 전체화면 가능)
- 추첨 결과 기록 및 내보내기

## 10. 제약 사항 및 리스크

| 리스크 | 영향 | 대응 방안 |
|--------|------|-----------|
| GitHub API rate limit | 대량 저장소 수집 시 지연 | 인증 토큰 사용 (5,000 req/hr), 수집 데이터 캐싱 |
| Claude API 응답 일관성 | 동일 프로젝트 재평가 시 점수 변동 | temperature 0 설정, 상세 루브릭으로 편차 최소화 |
| 배포 URL 접근 실패 | 스크린샷 캡처 불가 | 배포 보너스 0점 처리, 사유 기록, 관리자에게 알림 |
| private 저장소 접근 불가 | 평가 불가 | 제출 시 public 여부 사전 검증, 안내 메시지 |
| 토큰 한도 초과 | 대규모 프로젝트 분석 불완전 | 파일 우선순위 기반 선별적 포함, 요약 전략 |
| 참가자 사칭 제출 | 타인 명의로 제출 가능 | 사번 기반 간이 인증, 필요 시 관리자 검토 |
